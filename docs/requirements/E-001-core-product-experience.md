# E-001 核心產品體驗建立

## 背景與問題
- 這份 Epic 用來承接產品第一版的核心體驗需求。
- 所有來自 kickoff、會議紀錄或使用者訪談的功能性需求，都應先進入這裡整理，再決定哪些能升級到 PRD。

## 目標與非目標

### 目標
- 提供產品第一版的需求母題與需求邊界。
- 讓初期需求有共同歸屬，避免直接把討論內容寫進 PRD。

### 非目標
- 本 Epic 不直接等於 PRD。
- 本 Epic 不直接定義技術設計或 implementation plan。

## 當前決策
- 第一版產品需求先由此 Epic 承接。
- 子需求編號格式固定為 `E-001-xxx`。
- 只有已定案、範圍具體、可驗收的子需求才可升級進 PRD。

## Phase 1
- 整理 kickoff 與初期討論中的核心需求。
- 收斂第一版使用者體驗與 scope。

## Phase 2
- 將已定案需求逐步升級到 PRD。
- 補齊驗收方式、限制條件與例外情境。

## Phase 3
- 將已完成與已驗收項目沉澱為穩定產品基線。

## Requirement List

### E-001-001 核心主流程可用
- 狀態：未開始
- 類型：已定案
- 第一版至少需要讓主要使用者完成最核心的主流程。
- 尚未進入 PRD / TDD。

### E-001-002 首屏或入口體驗清楚
- 狀態：未開始
- 類型：待釐清
- 使用者進入產品後，應該能快速理解產品目的與下一步行動。
- 尚未進入 PRD / TDD。

### E-001-003 第一版範圍邊界清楚
- 狀態：未開始
- 類型：已定案
- 需要明確列出第一版必做與不做，避免 scope 膨脹。
- 尚未進入 PRD / TDD。

### E-001-004 音檔轉錄後可生成會議記錄
- 狀態：進行中
- 類型：已定案
- 使用者在既有音檔轉錄流程完成後，可以直接產出結構化會議記錄。
- 第一版會議記錄固定包含：`摘要`、`重點討論`、`決議事項`、`待辦事項`。
- 會議記錄生成服務需支援 OpenAI-compatible API，部署目標可為真正的 OpenAI API 或 vLLM。
- 第一版沿用既有逐字稿結果，不新增獨立的音檔直出會議記錄流程。
- 進入 PRD / TDD：是。

## 未決問題
- 目前哪些需求已足夠清楚，可以正式進入 PRD。
- 第一版驗收的成功標準是否已足夠具體。
- OpenAI-compatible API 的模型名稱、base URL 與金鑰管理方式需在 TDD 落實。

## 進入 PRD 條件
- 子需求已定案。
- 對應產品行為與驗收方向清楚。
- 無重大策略衝突或依賴未解。
