- 對應需求：`E-001-004 音檔轉錄後可生成會議記錄`

## 1. 背景與目的
目前產品已能讓使用者上傳音檔並取得逐字稿，但會議型內容的最終需求通常不是原始轉錄文字，而是可立即分享、追蹤與執行的會議記錄。若使用者還需要手動把逐字稿貼到其他 LLM 工具整理，流程會中斷，且不同工具的摘要格式不一致。

本次需求要在既有轉錄流程後提供一個內建的會議記錄生成能力，讓使用者在同一個產品流程中從音檔一路完成到結構化會議記錄。生成服務需採 OpenAI-compatible API，部署時可接真正的 OpenAI API 或 vLLM，相同產品流程不應綁死單一供應商。

## 2. 產品目標
- **G1**：使用者在現有轉錄流程完成後，可以一鍵生成固定格式的會議記錄。
- **G2**：相同功能可透過 OpenAI-compatible API 接到不同推論服務，不改動使用者流程。
- **G3**：會議記錄內容足夠穩定，能直接用於閱讀、複製與下載。

## 3. 目標使用者與情境
- 受眾：需要整理訪談、內部會議、客戶會議或專案同步錄音的使用者。
- 使用情境：使用者上傳會議音檔完成轉錄後，希望快速得到可閱讀的會議記錄，而不是自行整理逐字稿。
- 核心任務：從音檔產出逐字稿與結構化會議記錄，縮短整理與追蹤成本。

## 4. 核心體驗概念
- 第一版會議記錄是既有轉錄流程的延伸能力，不是新的獨立工作流。
- 會議記錄使用固定輸出結構，降低格式漂移並提升可讀性。
- 生成步驟應清楚顯示正在處理、成功完成或失敗原因。

## 5. 範圍（Scope）

### 5.1 In Scope（MVP 必做）
1. 轉錄結果頁可對既有逐字稿發起「生成會議記錄」操作。
   - 使用者會看到明確的操作入口。
   - 使用者能在轉錄完成後觸發會議記錄生成。
   - 完成條件是前端能成功呼叫後端並取得會議記錄內容。
2. 後端提供會議記錄生成 API。
   - API 以逐字稿文字為輸入，而非直接重新處理音檔。
   - API 會呼叫 OpenAI-compatible chat/completions 類型服務。
   - 完成條件是後端能回傳固定欄位的會議記錄結果。
3. 第一版會議記錄固定輸出四個區塊。
   - 使用者會看到 `摘要`、`重點討論`、`決議事項`、`待辦事項`。
   - 使用者能閱讀、複製或下載會議記錄。
   - 完成條件是四個區塊在成功回應後都能穩定呈現。
4. 生成失敗時有清楚回饋。
   - 使用者會看到設定缺失、API 失敗或格式異常的錯誤訊息。
   - 使用者能重新嘗試。
   - 完成條件是失敗不會破壞既有轉錄結果，且錯誤可辨識。

### 5.2 Out of Scope（MVP 不做）
- 不做音檔直接生成會議記錄的獨立流程。
- 不做自訂會議記錄模板或欄位排序。
- 不做多人協作、評論、標註或任務同步到外部工具。
- 不做會議記錄歷史版本管理。
- 不保證不同模型供應商輸出內容完全一致，只要求符合固定區塊格式。

## 6. 使用者流程
1. 使用者進入首頁並上傳音檔。
2. 系統完成既有轉錄流程，顯示逐字稿結果。
3. 使用者在轉錄結果區看到「生成會議記錄」入口並觸發。
4. 系統顯示生成中狀態。
5. 成功時顯示四個固定區塊的會議記錄內容，使用者可複製或下載。
6. 失敗時保留逐字稿結果並顯示錯誤訊息，使用者可重試。

## 7. 驗收標準
- 轉錄成功後，使用者可以在同一頁面對該逐字稿生成會議記錄。
- 後端可透過可配置的 OpenAI-compatible API base URL、model，以及在供應商需要時使用的 API key 完成請求。
- 成功結果至少包含 `摘要`、`重點討論`、`決議事項`、`待辦事項` 四個欄位，前端有對應顯示。
- 生成失敗時，逐字稿與既有下載/複製能力仍可使用。
- 若缺少會議記錄 API 設定，系統能回傳明確錯誤，不得 silently fail。

## 9. 非功能需求（NFR）
- 效能：會議記錄生成屬附加步驟，不應阻塞既有轉錄完成流程。
- 可用性：若未設定會議記錄 API，UI 需顯示可理解的失敗原因。
- 穩定性：外部 LLM 呼叫失敗時，既有轉錄結果不可遺失。
- 安全性：若供應商需要 API key，金鑰僅保存在伺服器端環境變數，不在前端暴露。

## 10. 開放問題
- 第一版下載格式是否只提供純文字，或同時輸出 Markdown。
- 待辦事項是否要在 UI 層再切出欄位化格式；MVP 先接受文字條列。
