## 1. 背景與目的
目前產品已具備音檔轉錄相關實作，但正式需求文件一度偏向尚未落地的延伸功能，導致規格與現況不一致。

本次 PRD 先把第一版 MVP 收斂為一條明確主流程：使用者上傳音檔後，可以在同一個產品流程中取得逐字稿文字與 SRT 字幕，並能複製、下載與查看處理狀態。主流程穩定可用後，產品也應支援從既有逐字稿進一步生成結構化會議記錄，讓使用者不必切換到其他工具整理摘要與待辦。

## 2. 產品目標
- **G1**：使用者可以上傳單一音檔並成功取得逐字稿文字。
- **G2**：使用者可以取得對應的 SRT 字幕結果，並能複製或下載。
- **G3**：轉錄流程對使用者而言可被理解，包含處理中、完成、失敗等狀態回饋。
- **G4**：使用者可在逐字稿完成後直接生成結構化會議記錄。
- **G5**：維護者依照正式文件操作時，可用單一 JavaScript 套件管理工具完成前端安裝、啟動與建置。
- **G6**：新進維護者打開 `README.md` 時，可在同一份入口文件中快速理解產品主流程、開發模式、production backend 設定與正式 spec 入口。

## 3. 目標使用者與情境
- 受眾：需要快速把音檔轉成可閱讀文字或字幕的使用者。
- 使用情境：使用者手上已有音檔，希望直接在網頁上傳後取得逐字稿，並在需要時立刻整理成會議記錄，不依賴額外 CLI 或後處理工具。
- 核心任務：從音檔取得 TXT / SRT 結果，並在同一流程中產出可閱讀的會議記錄。

## 4. 核心體驗概念
- MVP 先聚焦在「音檔 -> 逐字稿 / 字幕」這條主流程。
- 主流程完成後可直接從既有逐字稿生成會議記錄，避免切換工具。
- 同頁完成上傳、等待、查看結果、複製、下載與延伸整理。
- 繁體轉換可保留為現有輔助能力，但不是定義 MVP 是否成立的唯一條件。

## 5. 範圍（Scope）

### 5.1 In Scope（MVP 必做）
1. 使用者可上傳支援的音檔並建立轉錄任務。
   - 使用者會看到檔案選擇或拖放入口。
   - 使用者可選擇語言，並送出轉錄請求。
   - 完成條件是系統建立背景任務並回傳可查詢的 task id。
2. 使用者可追蹤轉錄進度並得知最終結果。
   - 使用者會看到處理中、完成或失敗等狀態。
   - 完成條件是前端可透過 task 狀態 API 驅動結果畫面。
3. 使用者可取得逐字稿 TXT 與 SRT 字幕。
   - 使用者會看到文字結果與字幕結果。
   - 使用者能複製或下載結果。
   - 完成條件是成功結果包含可用的 TXT / SRT 內容。
4. 使用者可在逐字稿完成後生成結構化會議記錄。
   - 使用者會在結果區看到明確的會議記錄生成入口。
   - 第一版固定輸出 `摘要`、`重點討論`、`決議事項`、`未決議事項`、`待辦事項` 五個區塊。
   - `未決議事項` 與 `待辦事項` 必須語意分離；前者描述尚未拍板議題，後者描述已有後續執行責任的行動。
   - 會議記錄整理稿由 backend 生成，前端不以本地規則直接產生正式結果。
   - 使用者可把整理稿匯出為 `Markdown` 與 `DOCX`。
   - 匯出檔名必須帶原始音檔名與時間戳，以提高辨識度並避免撞名。
   - 完成條件是系統以既有逐字稿內容產出可讀的會議記錄結果，且生成服務支援 OpenAI-compatible API。
5. 維護者可依文件使用 `npm` 完成 JavaScript 相關工作流。
   - 維護者會在 README、agent 指引與自動化設定看到一致的 `npm` 指令。
   - 完成條件是 repo 不再把 `pnpm` 視為正式指令來源，且 npm workspace 範圍與實際 Node 專案一致。
6. 維護者可透過 README 快速取得正確的專案入口資訊。
   - README 需清楚說明目前產品核心流程是「音檔轉錄 -> TXT / SRT -> 會議記錄」。
   - README 需區分快速開始、開發模式、production backend 與正式文件入口。
   - README 中提到的 repo 結構、檔名與指令需以實際現況為準，不保留易與實作漂移的錯誤內容。
   - 完成條件是新進維護者只讀 README 即可找到主要能力、正確開發指令與 `docs/prd.md` / `docs/tdd.md` / `docs/requirements/README.md` 入口。

### 5.2 Out of Scope（目前不作為 MVP 完成條件）
- 不把影音轉音訊工具視為本 MVP 成立與否的必要條件。
- 不做多使用者歷史保存、任務跨重整持久化或長期結果儲存保證。
- 不做複雜模型參數管理、自訂 prompt、或進階後製編輯流程。
- 不提供獨立於逐字稿之外的「音檔直接生成會議記錄」新流程。
- 不支援第一版之外的自訂會議記錄段落模板。

## 6. 使用者流程
1. 使用者進入首頁並選擇音檔。
2. 使用者送出轉錄請求。
3. 系統建立背景任務並回傳 task id。
4. 前端持續查詢任務狀態，直到完成或失敗。
5. 成功時顯示 TXT / SRT 結果，使用者可複製、下載，或在需要時轉成繁體中文。
6. 使用者在結果區觸發會議記錄生成。
7. 前端把既有逐字稿送到 backend 會議記錄生成接口。
8. backend 經由 OpenAI-compatible provider 回傳結構化會議記錄，前端顯示在同頁結果區。
9. 使用者可將整理稿下載為 `Markdown` 或 `DOCX`，檔名帶原始音檔名與時間戳。

## 7. 驗收標準
- 使用者能從首頁完成一次音檔上傳並成功取得逐字稿。
- 成功結果至少包含 TXT 與 SRT 兩種可讀輸出。
- 使用者能清楚看到任務處理中的狀態，完成後能讀取結果。
- 使用者能複製或下載至少一種結果格式。
- 使用者能在逐字稿完成後生成會議記錄，且結果固定包含 `摘要`、`重點討論`、`決議事項`、`未決議事項`、`待辦事項`。
- 若逐字稿內同時存在未拍板議題與後續執行事項，系統需能分別落在 `未決議事項` 與 `待辦事項`，不可全部塞進同一區塊。
- 會議記錄結果需來自 backend 正式生成接口，而不是純前端規則式假資料。
- 使用者能下載會議記錄整理稿為 `Markdown` 與 `DOCX`，且檔名帶原始音檔名與時間戳。
- 會議記錄生成能力必須使用既有逐字稿資料，不新增獨立的音檔直出分支流程。
- 若轉錄失敗，UI 與 API 至少要能提供可辨識的失敗狀態。
- README 需描述目前正式產品能力與維護入口，且其中指令、檔名與文件連結需可對應 repo 實際內容。

## 9. 非功能需求（NFR）
- 效能：轉錄採背景任務執行，不應阻塞前端請求回應。
- 可用性：首頁主流程需讓使用者知道下一步是上傳並等待結果。
- 穩定性：任務完成後，成功結果需可被前端正確讀取並顯示。
- 安全性：前端僅經由既有 API proxy 呼叫後端，不直接暴露後端內部設定。
- 相容性：會議記錄生成服務需支援 OpenAI-compatible API，以便對接 OpenAI API 或 vLLM 類型部署。
- 可維護性：JavaScript 安裝、啟動與建置文件需統一使用 `npm`，避免不同套件管理工具並存造成操作歧義。

## 10. 開放問題
- 任務狀態與結果讀取的錯誤 contract 是否已足夠穩定，仍需後續驗證。
- 影音轉音訊工具是否要在下一輪正式納入同一份產品 PRD。
